## 2. 单K线异常移动 → 强平+冻结机制

**依赖**: 任务1完成后,从历史数据反推合理阈值。

**目标**: 在"流动性熔断"之外增加一层更严格的保护——单根M1 K线波动超过某个阈值时,**立即强平所有持仓 + 冻结EA**(不再开新单)。

**与现有机制的区别**:

| 机制 | 阈值 | 行为 |
|------|------|------|
| 流动性熔断(现有) | M1波动 ≥ $10 | 只拦L1, 加仓正常, 5min后恢复 |
| **新黑天鹅熔断** | M1波动 ≥ ?(待定) | **全平+永久冻结**(直到人工解锁) |
| 50%总亏强平 | equity < balance×50% | 全平,周期归零,但可继续 |

**待设计**:
- [ ] 阈值参数 `Inp_BlackSwanRange`: 单根M1 high-low(USD)
  - 建议初值: 根据任务1的历史数据,取过去3年第99百分位
  - 候选范围: $15-25 (覆盖大部分新闻冲击的初震)
- [ ] 触发后行为:
  - [ ] 立即 `CloseAllPositions()`
  - [ ] 设置全局标志 `g_emergencyFrozen = true`
  - [ ] `CheckEntry()` 和 `CheckMartingaleAdd()` 都返回
  - [ ] JSON导出新字段 `emergency_frozen: true`
  - [ ] 面板显示醒目"❗已冻结"
- [ ] 解锁机制:
  - [ ] 选项A: 完全手动(EA重启或手动改参数)
  - [ ] 选项B: N小时后自动恢复
  - [ ] 选项C: 价格回稳后(后续X根K线波动均<Y)自动恢复
  - 推荐: **选项A**(黑天鹅事件需人工评估市场)
- [ ] 日志:
  - [ ] `[黑天鹅] M1波动$X 已超过$Y,全平+冻结!`
  - [ ] 平仓时记录每张订单盈亏,导出到 trades.jsonl 时标记 `close_reason: "BLACK_SWAN"`

**设计原则**:
- 比50%强平更早介入,避免马丁在剧烈行情中被打到极限
- 比流动性暂停更严厉,因为单K线$15+不是普通新闻波动,可能是闪崩/暴涨
- 冻结需要人工评估,防止EA在异常市场中"自动重新交易"导致连续损失

**实现优先级**: 等任务1的统计数据出来再定阈值,代码框架可以先准备。

## 3. 距离序列 vs 手数序列的数学耦合 — [x] 已排除误报

**⚠️ 之前的问题描述基于错误手算,实际数值模拟已证明基线序列数学正确**

### 3.1 正确公式

定义 `lag = 平均成本 - 当前价 = 浮亏 / (累计手数 × 100)`

递推关系:
```
new_lag = (cum_{N-1} / cum_N) × (old_lag + Δ_N)
```

篮子TP需要的反弹:
```
反弹 = (lag + TP_basket) / (cum × 100)
```

### 3.2 正确的反弹要求 (基线序列, TP_basket=$8)

| 层 | Δ | 累计手数 | lag | 反弹要求 |
|----|---|---------|-----|--------|
| L1 | - | 0.01 | 0 | $8.00 |
| L2 | 4 | 0.02 | 2 | $5.00 |
| L3 | 5 | 0.04 | 3.5 | $2.88 |
| L4 | 6 | 0.06 | 6.3 | $2.39 |
| L5 | 7 | 0.09 | 8.9 | $1.88 |
| L6 | 8 | 0.13 | 11.7 | $1.52 |
| L7 | 9 | 0.18 | 14.9 | $1.28 |
| L8 | 10 | 0.25 | 18.0 | $1.04 |
| L9 | 11 | 0.35 | 20.7 | $0.82 |
| L10 | 12 | 0.47 | 24.3 | $0.69 |
| L11 | 14 | 0.60 | 30.0 | $0.63 |

**结论**: 反弹要求全程单调递减,完全符合马丁"越深越易TP"的核心机制。

### 3.3 实盘验证

85笔真实交易中,深层周期(L5-L7)全部盈利:
- L7 @ 06-24: 盈利 $10.19 (反弹约$1.4,匹配理论$1.28)
- L7 @ 06-30: 盈利 $10.22
- 无一次触发L8+

**基线序列 (Δ: 4,5,6,7,8,9,10,11,12,14 + 手数: 0.01,0.01,0.02,0.02,0.03,0.04,0.05,0.07,0.10,0.12,0.13) 保持不动。**

## 4. V2 策略双引擎设计

### 4.1 问题背景

V1 生产环境两大痛点:
1. **浅层收益不足** — L1/L2 平均 $5, 中轨TP $2 频繁触发
2. **日均单量偏少** — 5-10 笔/日, 大量时间空仓等信号

### 4.2 解决方向

**问题1 (收益低)** → V2 分层TP + 手数序列调整 (详见 [ea/布林带马丁V2/STRATEGY_v2.md](ea/布林带马丁V2/STRATEGY_v2.md))

**问题2 (单量少)** → 新增第二引擎, 双引擎并行

### 4.3 双引擎设计

**引擎A (反转引擎, 现有逻辑)**
```
触发: BB 触轨 + RSI 极值 (28/72)
方向: 逆势 (触上轨做空, 触下轨做多)
TP: 分层 (V2设计)
```

**引擎B (顺势引擎, 新增)**
```
触发: 价格回到 BB 中轨 (从外轨回归后)
方向: 参考 M30 大势
       M30 上半区 → 做多
       M30 下半区 → 做空
       M30 中性区 → 不触发
TP: 期望收益减半, 快进快出
```

### 4.4 待决策清单 (关键选择)

- [ ] **优先级1: 冲突处理**
  - 引擎A/B 是否允许同时持仓?
  - 推荐: 互斥, B 只在完全空仓时触发

- [ ] **优先级1: "回中轨"判定**
  - A. 价格瞬间穿越中轨
  - B. 价格在中轨±K×ATR 附近
  - C. 从外轨回归到中轨 (推荐)

- [ ] **优先级2: M30 方向判定**
  - 复用现有 `CheckM30TrendFilter` 位置式 (推荐)
  - 或用 M30 中轨斜率

- [ ] **优先级2: 引擎B 手数序列**
  - 用引擎A 同一套 (激进)
  - 缩小到 0.01-0.05, 累计约 0.3手 (推荐, 主辅分工)

- [ ] **优先级2: 引擎B 出场逻辑**
  - 篮子TP 减半 ($4-13)
  - 中轨TP 不适用 (自身在中轨启动)
  - 是否需要"反向触外轨"作为止盈信号?

- [ ] **优先级3: 参数区分**
  - 引擎B 是否独立的 MagicNumber?
  - 报表中如何区分两个引擎的贡献?

### 4.5 风险评估

**潜在风险**:
1. 引擎B 触发时若逆势, 可能陷入更深马丁
2. 单量翻倍后, 若胜率下降, 反而伤害总盈利
3. M30 判定可能滞后, 顺势入场时趋势已尾声

**缓解措施**:
1. 引擎B 手数序列缩小, 降低失败成本
2. 引擎B 加严格过滤: M30 必须明确方向 (非中性区) + 波动率过滤
3. 前 2 周并行观察, 单独统计 B 的胜率

### 4.6 实施顺序

1. [ ] 完成 V2 分层TP 单引擎版本 (先解决问题1)
2. [ ] V2 单引擎跑 1 周, 确认盈利效果
3. [ ] 决策清单达成一致后, 加入引擎B
4. [ ] V2 双引擎并行 V1 生产, 对比数据
